## SECTION 1: Core Concepts — Key Terms & Definitions

### OWASP & Framework
| Term | Definition |
|------|------------|
| **OWASP Top Ten** | An industry-standard awareness document listing the ten most critical security risks to web applications, representing broad expert consensus. |
| **OWASP ASVS** | Application Security Verification Standard — a framework providing three levels of security verification guidance for web applications. |
| **A01: Broken Access Control** | The top OWASP category covering failures in authentication, authorization, session management, data transit protection, SSRF, and CSRF. |
| **A02: Security Misconfiguration** | Vulnerabilities arising from improperly configured security settings in applications, frameworks, or infrastructure. |
| **A03: Software Supply Chain Failures** | Risks from using obsolete, unmaintained, or vulnerable third-party components and dependencies. |
| **A04: Cryptographic Failures** | Vulnerabilities related to weak, improper, or absent cryptographic protections (previously "Sensitive Data Exposure"). |
| **A05: Injection** | Attacks where untrusted data is sent to an interpreter as part of a command or query (e.g., SQL injection, XSS). |

### Authentication
| Term | Definition |
|------|------------|
| **Authentication** | The process of verifying the identity of a user — confirming that the entity is who it claims to be. |
| **Authorization** | The process of determining what an authenticated user is permitted to do — enforcing access control on resources and actions. |
| **Session Management** | The mechanism that links multiple HTTP requests to a specific authenticated user, bridging authentication and authorization across stateless connections. |
| **MFA / 2FA** | Multi-Factor / Two-Factor Authentication: requiring proof of identity from two or more independent categories (knowledge, possession, inherence). |
| **Single Sign-On (SSO)** | An authentication scheme allowing users to authenticate once with an identity provider (e.g., Google, Facebook) and gain access to multiple services without re-entering credentials. |
| **TOTP** | Time-based One-Time Password — a short-lived numeric code generated by an authenticator app (e.g., Google Authenticator) using a shared secret and the current timestamp. |
| **Brute Force Attack** | A method of compromising credentials by systematically trying every possible combination of passwords. |
| **CAPTCHA** | A challenge-response test designed to distinguish human users from automated bots, used to mitigate brute force attacks. |
| **Hard-coded Credentials** | Credentials (usernames/passwords) embedded directly in source code or configuration files — a critical security anti-pattern. |

### Authorization
| Term | Definition |
|------|------------|
| **RBAC** | Role-Based Access Control — a policy where permissions are granted based on roles (e.g., admin, viewer) assigned to users, rather than to individuals directly. |
| **ABAC** | Attribute-Based Access Control — a generalized model that evaluates multiple attributes of the user, resource, environment, and action to make authorization decisions. |
| **Deny by Default** | An authorization design principle where access is denied unless explicitly permitted; all unspecified access is automatically blocked. |
| **OAuth2** | An open authorization framework allowing third-party applications to access user resources (e.g., profile, contacts) without receiving the user's credentials. |
| **OpenID Connect (OIDC)** | An identity layer built on top of OAuth2 that allows applications to verify user identity via an ID Token (JWT format) returned alongside the OAuth2 access token. |
| **JWT (JSON Web Token)** | An open standard (RFC 7519) for securely transmitting claims between parties as a Base64Url-encoded, digitally signed JSON object with three parts: Header, Payload, and Signature. |
| **Refresh Token** | A long-lived credential used in OAuth2 to obtain a new access token automatically when the current one expires, maintaining user session continuity. |
| **XACML** | eXtensible Access Control Markup Language — an XML-based, OASIS-standard language for expressing ABAC policies. |
| **Casbin** | An open-source authorization library supporting RBAC, ABAC, and other access control models. |

### Session Management
| Term                          | Definition                                                                                                                                          |
| ----------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Session Identifier**        | A unique, unpredictable token (minimum 128 bits, generated from a PRNG) that maps a browser session to a specific authenticated user on the server. |
| **HttpOnly Cookie Flag**      | A cookie attribute preventing JavaScript from accessing the cookie, mitigating XSS-based session token theft.                                       |
| **Secure Cookie Flag**        | A cookie attribute instructing the browser to transmit the cookie only over HTTPS connections.                                                      |
| **SameSite Cookie Attribute** | Controls whether a cookie is sent with cross-site requests; values include `Strict`, `Lax`, and `None`. Mitigates CSRF.                             |
| **Session Timeout**           | The automatic invalidation of an inactive session after a configured period of inactivity.                                                          |

### Data in Transit
| Term | Definition |
|------|------------|
| **HTTPS** | HTTP over TLS — encrypts HTTP communication between browser and server to prevent eavesdropping, session hijacking, and man-in-the-middle attacks. |
| **TLS** | Transport Layer Security — a cryptographic protocol providing communication security over a network; the modern successor to SSL. |
| **SSL** | Secure Sockets Layer — the deprecated predecessor to TLS. Still commonly referenced but should not be used. |
| **Certificate Authority (CA)** | A trusted third party that issues and validates digital certificates, providing assurance that a server's public key truly belongs to the claimed domain/organization. |
| **Public Key Certificate** | A digitally signed document binding a public key to an entity (domain, organization), used to authenticate servers in TLS. |
| **HSTS** | HTTP Strict Transport Security — an HTTP response header instructing browsers to only communicate with the server over HTTPS for a specified duration. |
| **DV / OV / EV Certificates** | Three tiers of CA validation: Domain Validation (cheapest; verifies domain ownership only), Organization Validation (verifies organization identity), Extended Validation (most rigorous; highest trust). |
| **Man-in-the-Middle (MitM)** | An attack where an adversary secretly intercepts and potentially alters communications between two parties who believe they are communicating directly. |
| **Cache-Control Header** | HTTP response header controlling caching behavior: `no-store` prevents any caching; `no-cache` requires revalidation before using cached content; `private` restricts caching to the end user only. |

### Attacks: XSS, CSRF, SSRF
| Term                                   | Definition                                                                                                                                                                         |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **XSS (Cross-Site Scripting)**         | An injection attack where malicious scripts are injected into a trusted website and executed in the victim's browser, potentially stealing session data or impersonating the user. |
| **Reflected XSS**                      | A non-persistent XSS attack where the malicious script is embedded in a URL, reflected by the server in the response, and executed immediately in the browser.                     |
| **Stored (Persistent) XSS**            | An XSS attack where the malicious script is permanently stored on the server (e.g., in a database via a comment field) and executed every time a victim loads the affected page.   |
| **DOM-based XSS**                      | An XSS variant where the attack payload modifies the DOM environment in the victim's browser, executed entirely client-side without server reflection.                             |
| **CSP (Content Security Policy)**      | An HTTP response header that instructs the browser to only load resources (scripts, images, etc.) from explicitly whitelisted origins, reducing XSS risk.                          |
| **CSRF (Cross-Site Request Forgery)**  | An attack where a malicious page tricks a victim's authenticated browser into sending an unintended, authenticated request to a trusted web application.                           |
| **SSRF (Server-Side Request Forgery)** | An attack where a malicious input causes the server itself to make unintended HTTP requests to internal services, localhost, or cloud metadata endpoints.                          |

### Password Storage
| Term | Definition |
|------|------------|
| **Password Hashing** | A one-way cryptographic transformation of a password into a fixed-length digest; the original password cannot be directly recovered from the hash. |
| **Salt** | A unique, randomly generated value added to a password before hashing to ensure that identical passwords produce different hashes and to defeat precomputed (rainbow table) attacks. |
| **Work Factor / Iterations** | A configurable parameter in modern hash algorithms (bcrypt, Argon2) that controls computational cost, making brute force attacks more expensive. |
| **Argon2id** | The current OWASP-recommended password hashing algorithm (post-2019), designed to be memory-hard and resistant to GPU-accelerated brute force. |
| **bcrypt / scrypt** | Widely used, computationally expensive password hashing algorithms that are considered acceptable but are superseded by Argon2id for new systems. |
| **MD5 / SHA-1** | Cryptographic hash functions now considered broken for password storage — computationally cheap to brute force, especially with GPU acceleration. |
| **Rainbow Table** | A precomputed table of hash values used to reverse hash functions; salting defeats this attack. |

### SQL Injection / Database Security
| Term | Definition |
|------|------------|
| **SQL Injection** | An injection attack where user-supplied input is interpreted as SQL commands, allowing attackers to manipulate database queries. |
| **Parameter Binding** | A technique that separates SQL code from data values using placeholders, preventing user input from being interpreted as executable SQL. |
| **Stored Procedure** | A precompiled database routine; does NOT inherently prevent SQL injection unless parameter binding is used within the procedure. |
| **Blind SQL Injection** | SQL injection where the attacker cannot see direct query results but infers information from application behavior (e.g., true/false responses, timing differences). |
| **Nonblind SQL Injection** | SQL injection where the attacker receives direct, informative feedback from the server (e.g., error messages revealing data). |
| **File Upload Vulnerability** | An injection-class vulnerability where an attacker uploads an executable script instead of a legitimate file (e.g., image), which the server may then execute. |

## SECTION 2: Mechanisms & How Things Work

### 2.1 — TLS/HTTPS Handshake (How Certificates Work)
1. **Server** presents its public key certificate to the client.
2. **Client** verifies the certificate against a trusted CA's root certificate.
3. If verified, client uses the server's **public key** to establish a secure symmetric session key.
4. All subsequent communication is encrypted with that symmetric key.
5. **CA hierarchy**: Root CA → Intermediate CA → End-entity certificate (signed by private key of issuer).

**Why CA matters**: Without a trusted CA, a server cannot prove its identity — any server could claim to be "bank.com". CAs provide the chain of trust.

### 2.2 — OAuth2 Authorization Flow
1. User initiates login via a third-party provider (e.g., Google).
2. Application redirects user to the OAuth2 **Authorization Server**.
3. User authenticates with the provider and **grants access** to specific scopes.
4. Authorization Server returns an **Authorization Code** to the application.
5. Application exchanges the code for an **Access Token** (and optionally a **Refresh Token**) via a back-channel server-to-server request.
6. Application uses the Access Token to call protected resource APIs on behalf of the user.
7. When the Access Token expires, the **Refresh Token** is used to obtain a new one automatically.

**OIDC Extension**: If OIDC is used, the Authorization Server also returns an **ID Token** (a JWT) containing verified claims about the authenticated user's identity.

### 2.3 — JWT Structure & Verification
A JWT has three Base64Url-encoded parts separated by dots: `header.payload.signature`

- **Header**: Algorithm (`alg`) and token type (`typ`).
- **Payload**: Claims — `sub` (subject/user ID), `name`, roles, `iat` (issued-at), `exp` (expiry), etc.
- **Signature**: `HMAC-SHA256(base64(header) + "." + base64(payload), secret)` — proves the token has not been tampered with.

**Critical**: The payload is **encoded, not encrypted**. Anyone can decode it. The signature only guarantees **integrity**, not confidentiality.

**JWT Vulnerabilities**:
- Server does not verify signature → attacker can forge any claims.
- `alg: none` attack → attacker sets algorithm to "none", stripping the signature.
- Weak/trivial secret → offline brute force of the HMAC key.

### 2.4 — Password Hashing Process (Correct Flow)
**Registration:**
1. User submits plain-text password.
2. System generates a unique, cryptographically random **salt** (≥ 32 bits per OWASP; ≥128 bits per NIST).
3. System hashes `salt + password` using a strong algorithm (Argon2id).
4. System stores: `{salt, hash, work_factor}` — **never** the plain-text password.

**Login:**
1. User submits plain-text password.
2. System retrieves stored `salt` for that user.
3. System hashes `salt + submitted_password` using the stored work_factor.
4. System compares the resulting hash to the stored hash.
5. Match → authenticated; no match → rejected.

### 2.5 — SQL Injection via String Concatenation

**Vulnerable pattern:**
```sql
SELECT * FROM users WHERE name = '" + userInput + "'
```
If `userInput = "'; DROP TABLE users; --"`, the query becomes destructive SQL.

**Safe pattern (Parameter Binding):**
```sql
SELECT * FROM users WHERE name = ?  -- placeholder
```
The driver treats the user input as a **data value only**, never as executable SQL syntax.

### 2.6 — XSS Attack Mechanisms

| XSS Type  | Attack Vector                     | Server Involvement          | Defense Location                               |
| --------- | --------------------------------- | --------------------------- | ---------------------------------------------- |
| Reflected | Malicious URL sent to victim      | Server reflects input back  | Server-side input validation                   |
| Stored    | Malicious script stored in DB     | Server serves stored script | Server-side input validation + output encoding |
| DOM-based | Malicious URL modifies DOM via JS | None (client-only)          | Client-side input sanitization                 |

**Prevention**: Escape and sanitize all untrusted data **before** rendering. Use CSP headers to restrict script sources.

### 2.7 — RBAC vs. ABAC Decision Logic

**RBAC Decision**: `if user.role == "admin" → permit delete`
- Roles are assigned to users; permissions are assigned to roles.
- Simple, predictable, maintainable when permissions are relatively static.

**ABAC Decision**: `if user.department == "HR" AND resource.classification == "HR_Only" AND time.isWorkHours() → permit read`
- Policies evaluate multiple attributes simultaneously.
- Highly flexible; supports dynamic, context-sensitive decisions.
- Standard policy format: XACML (XML-based).

## SECTION 3: Comparisons & Trade-offs

### 3.1 — XSS vs. CSRF vs. SSRF

| Aspect | XSS | CSRF | SSRF |
|--------|-----|------|------|
| **Full name** | Cross-Site Scripting | Cross-Site Request Forgery | Server-Side Request Forgery |
| **Actor** | Victim's browser runs attacker JS | Victim's browser sends request | Server sends request |
| **Attack target** | User's browser/session/data | Web app's trust in user session | Internal services, cloud metadata |
| **Typical impact** | Session hijack, data theft, page defacement | Perform actions as victim | Access internal systems, expose credentials |
| **Requires auth?** | No (targets any visitor) | Yes (exploits authenticated session) | No (exploits server-side functionality) |
| **Prevention** | Output encoding, CSP | CSRF tokens, SameSite cookies | Input validation, allowlists, network segmentation |

### 3.2 — RBAC vs. ABAC

| Dimension | RBAC | ABAC |
|-----------|------|------|
| **Complexity** | Low — simple role assignment | High — multi-attribute policy evaluation |
| **Flexibility** | Low — coarse-grained | High — fine-grained, dynamic |
| **Scalability of permissions** | Limited (role explosion problem) | Scales well with attribute diversity |
| **Standard** | Common in most frameworks | XACML (OASIS standard) |
| **Use when** | Permissions are static; roles map cleanly to domain roles | Dynamic conditions (time, location, resource attributes); existing HR/profile attributes |
| **Tools** | Spring Security, CanCanCan | Casbin, OPA (Open Policy Agent) |

### 3.3 — Password Hashing Algorithms

| Algorithm | Security Level | Speed | Notes |
|-----------|---------------|-------|-------|
| MD5 | ❌ Broken | Very Fast | Do not use — trivially brute-forced with GPU |
| SHA-1 | ❌ Broken | Fast | Do not use for passwords |
| SHA-256 (plain) | ⚠️ Weak | Fast | Too fast; not designed for passwords |
| bcrypt | ✅ Acceptable | Slow (configurable) | Industry standard; memory-light |
| scrypt | ✅ Good | Slow, memory-hard | Better than bcrypt against GPU |
| **Argon2id** | ✅✅ Best | Slow, memory-hard | OWASP recommended post-2019 |

### 3.4 — CA Certificate Validation Levels

| Level | Validates | Cost | Trust Signal |
|-------|-----------|------|-------------|
| DV (Domain Validation) | Domain ownership only | Low | Basic padlock |
| OV (Organization Validation) | Organization identity + domain | Medium | Organization name visible |
| EV (Extended Validation) | Rigorous legal/organizational check | High | Highest trust |

### 3.5 — Cookie Attributes

| Attribute | Purpose | Security Benefit |
|-----------|---------|-----------------|
| `HttpOnly` | Prevents JS access to cookie | Mitigates XSS-based cookie theft |
| `Secure` | Sends cookie only over HTTPS | Prevents transmission over plain HTTP |
| `SameSite=Strict` | Blocks cookie in all cross-site requests | Strong CSRF protection |
| `SameSite=Lax` | Blocks cookie in cross-site POST | Moderate CSRF protection |
| `Domain` | Restricts cookie scope to domain/subdomain | Limits exposure |
| `Path` | Restricts cookie to specific URL path | Principle of least privilege |

### 3.6 — Authentication Methods

| Method | Factors | Pros | Cons |
|--------|---------|------|------|
| Username/Password | 1 (knowledge) | Simple | Phishable, reusable, breachable |
| SMS OTP | 2 (possession) | Familiar | SIM-swappable, interception risk |
| TOTP (Authenticator App) | 2 (possession) | More secure than SMS | Device loss risk |
| Hardware Token / Biometric | 2-3 | Strong | Cost, accessibility issues |
| SSO (OAuth2/OIDC) | Varies | Convenient, reduces credential sprawl | Single point of failure |

## SECTION 4: Common Misconceptions & Traps

> ⚠️ These are the concepts most likely to appear as **plausible but incorrect** answer choices.

1. **"Hiding UI elements = Authorization"** — FALSE. Removing a "Delete" button from the UI is a UX concern only. The server MUST enforce authorization independently on every request. Security through obscurity is not security.

2. **"Stored Procedures prevent SQL Injection"** — FALSE. SPs only prevent injection if they also use parameter binding internally and avoid string concatenation. An SP that concatenates user input is still injectable.

3. **"ORM frameworks (Hibernate, ActiveRecord) automatically prevent SQL Injection"** — FALSE. ORMs can still construct unsafe queries if developers use raw query strings or string concatenation instead of the framework's parameterized query APIs.

4. **"OAuth2 authenticates users"** — MISLEADING. OAuth2 is an **authorization** framework (delegated resource access). **OpenID Connect (OIDC)** is the identity layer that adds **authentication** on top of OAuth2.

5. **"JWT payload is encrypted and private"** — FALSE. JWT payload is Base64Url-**encoded**, not encrypted. Anyone who intercepts or inspects the token can read its contents. JWTs are integrity-protected (signed), not confidential, unless encrypted (JWE).

6. **"Redirecting HTTP to HTTPS is sufficient to prevent MitM"** — FALSE. The initial HTTP request before the redirect is still unprotected. HSTS eliminates this window by making browsers issue only HTTPS requests from the start.

7. **"CSRF attacks require JavaScript"** — FALSE. CSRF exploits the browser's automatic cookie inclusion in cross-origin requests. A simple hidden HTML form with `<form method="POST">` is sufficient — no JavaScript required.

8. **"Salting alone makes MD5/SHA-1 safe for passwords"** — FALSE. Salting defeats rainbow table attacks but does not overcome the fundamental problem that MD5 and SHA-1 are computationally cheap — they remain vulnerable to GPU-accelerated brute force even with salts.

9. **"NoSQL databases are immune to injection attacks"** — FALSE. MongoDB's `$where` operator, for example, can be exploited for injection. The principle of parameter binding applies to all database types.

10. **"CAPTCHA fully prevents brute force"** — FALSE. CAPTCHAs can be solved by humans hired by attackers (CAPTCHA farms), some are easier for machines than humans, and they create accessibility barriers.

11. **"Temporary account lockout fully prevents brute force without drawbacks"** — FALSE. Lockout can itself be weaponized as a Denial of Service attack against legitimate users, and it reveals to attackers that an account exists (undermining "conceal whether users exist").

12. **"DOM-based XSS is prevented by server-side validation"** — FALSE. DOM-based XSS occurs entirely in the client. Server-side validation does not see the DOM manipulation. Client-side sanitization is required.

## SECTION 5: High-Probability Exam Questions (MCQ)

---

**Q1.** A web application returns different error messages for "User not found" versus "Incorrect password." Which security principle does this violate?

- A) Deny by default
- B) Authorize on the server
- C) Conceal whether users exist ✅
- D) Preventing brute force attacks

**Explanation**: Returning distinct error messages for unknown usernames vs. wrong passwords enables **user enumeration** — attackers can build a valid user list. The correct approach is a generic message: "Incorrect user ID or password." (A) is about authorization defaults. (B) is about server-side enforcement. (D) is related but concerns rate limiting, not error message policy.

---

**Q2.** A developer uses Hibernate (an ORM) for all database access. The developer concludes that SQL injection is therefore impossible. This reasoning is:

- A) Correct — ORMs generate parameterized queries by default
- B) Correct — Hibernate uses binary protocols, not SQL
- C) Incorrect — ORMs prevent injection only if parameterized APIs are used consistently ✅
- D) Incorrect — Hibernate is vulnerable because it cannot use prepared statements

**Explanation**: ORMs do not guarantee SQL injection prevention. Developers can use raw SQL strings or string concatenation within the ORM, reintroducing the vulnerability. Parameter binding must be explicitly applied. (A) is false — ORMs allow raw queries. (D) is false — ORMs do support prepared statements.

---

**Q3.** Which JWT vulnerability allows an attacker to forge a valid-looking token by setting the algorithm field to "none"?

- A) Weak secret attack
- B) No-alg (algorithm confusion) attack ✅
- C) Replay attack
- D) Session fixation attack

**Explanation**: If the server accepts `"alg": "none"` in the JWT header, it skips signature verification entirely, accepting any payload the attacker supplies. (A) requires brute-forcing the HMAC secret. (C) involves reusing a valid token. (D) is a separate session management vulnerability.

---

**Q4.** A user submits input that is stored in the database and later displayed to other users. When victims view the page, the script runs in their browsers and exfiltrates their cookies. This describes:

- A) Reflected XSS
- B) DOM-based XSS
- C) CSRF
- D) Stored (Persistent) XSS ✅

**Explanation**: The defining feature of Stored XSS is that the malicious script is **persisted server-side** and delivered to all users who visit the affected page. (A) Reflected XSS is delivered via a URL and executed immediately — not stored. (B) DOM-based XSS is client-side only. (C) CSRF tricks the browser into sending requests, not executing injected scripts.

---

**Q5.** An application correctly enforces HTTPS for all pages. However, the HSTS header is not configured. A user types "http://example.com" in the browser. What risk remains?

- A) No risk — the server will redirect to HTTPS
- B) The server certificate may be invalid
- C) The initial HTTP request before the redirect is unprotected, enabling MitM ✅
- D) Cookies with the Secure flag will not be transmitted

**Explanation**: Without HSTS, the browser issues the first request over plain HTTP. An attacker performing MitM on that initial request can intercept or modify it before the redirect occurs. HSTS eliminates this gap by instructing the browser to always use HTTPS directly. (A) ignores the insecure initial request. (D) is false — Secure cookies won't be sent over HTTP, but the initial page request itself is exposed.

---

**Q6.** Which of the following BEST describes the role of a Certificate Authority (CA) in TLS?

- A) Encrypts all data transmitted between client and server
- B) Generates session keys for each TLS connection
- C) Acts as a trusted third party that vouches for the authenticity of a server's public key ✅
- D) Prevents man-in-the-middle attacks by monitoring network traffic

**Explanation**: The CA's function is to **sign certificates**, binding a public key to a verified identity. Clients trust the CA's signature as proof that the server is legitimate. (A) Encryption is done using keys negotiated during the TLS handshake — not by the CA. (B) Session keys are generated by the client/server during the handshake. (D) CAs do not monitor traffic.

---

**Q7.** A bank's web application logs the user out when the browser window is closed but does NOT invalidate the session server-side. What is the primary security risk?

- A) The session ID may be reused by a different user
- B) The session cookie persists and could be stolen and replayed ✅
- C) The user cannot log back in without reauthentication
- D) The application violates the Deny by Default principle

**Explanation**: Server-side session invalidation is mandatory. If only the client-side cookie is cleared (e.g., by closing the browser), the session token may still be valid on the server. An attacker who previously captured the session ID (e.g., via XSS or network sniffing) can still use it. (A) is a different risk (session fixation). (C) is incorrect — the user can log in again. (D) relates to authorization defaults, not session lifecycle.

---

**Q8.** A developer stores passwords using SHA-256 with a unique per-user salt. An attacker obtains the database. Which attack is MOST feasible?

- A) Rainbow table attack
- B) Dictionary / brute force attack ✅
- C) Birthday attack
- D) Side-channel attack

**Explanation**: Salting defeats precomputed rainbow tables (A). However, SHA-256 is designed for speed — attackers can compute billions of SHA-256 hashes per second using GPUs, making brute force/dictionary attacks feasible against weak passwords. The solution is to use a deliberately slow algorithm (Argon2id, bcrypt). (C) Birthday attacks target hash collision probability, not password recovery. (D) Side-channel attacks target hardware/timing, not hashed passwords.

---

**Q9.** An attacker sends a victim a link: `http://bank.com/transfer?amount=1000&to=attacker`. The victim is logged in. The link causes their browser to send the request with their session cookie. This is:

- A) Reflected XSS
- B) SSRF
- C) CSRF ✅
- D) Stored XSS

**Explanation**: CSRF exploits the browser's automatic inclusion of cookies in requests. The victim's authenticated session causes the bank to trust the forged request. No script injection occurs — just an unwanted authenticated request. (A) and (D) involve script injection. (B) SSRF is initiated by the server, not the victim's browser.

---

**Q10.** Which Cache-Control header value combination BEST protects sensitive authenticated responses from being stored by intermediate proxies or the user's browser?

- A) `public, max-age=3600`
- B) `private, max-age=3600`
- C) `no-cache`
- D) `private, no-cache, no-store` ✅

**Explanation**: `private` restricts caching to the end user's browser only (no shared proxies). `no-cache` requires revalidation with the server before serving cached content. `no-store` prevents any caching entirely. The combination of all three provides maximum protection for sensitive data. (A) allows public/shared proxy caching — dangerous. (B) allows browser caching. (C) alone only requires revalidation but permits storage.

## SECTION 6: One-Page Cheat Sheet — High-Priority Items

---

### 🔑 Authentication
- **Authentication** = Who are you? | **Authorization** = What can you do?
- **Session management** links the two across HTTP requests
- Use existing auth frameworks (Passport.js, etc.) — don't roll your own
- Require **reauthentication** for sensitive actions (e.g., password change, bank transfer)
- Return **generic error messages** — never reveal if username exists
- Mitigate brute force: **temporary lockout + CAPTCHA** (each has trade-offs)
- **Never hard-code credentials** in source code, Dockerfiles, or env files
- Support **MFA/2FA**: TOTP, hardware tokens, push notifications
- **SSO**: Convenient but single point of failure; always allow alternate login option

### 🔒 Authorization
- **Server-side enforcement is mandatory** — hiding UI elements is UX, not security
- **Deny by default**: if permission is not explicitly granted, access is denied
- Authorize **per resource** (files, profiles, APIs) — not just per page
- **RBAC**: roles → permissions; use when permissions are relatively static
- **ABAC**: attribute policies; use when context-dependent (time, department, resource type)
- **Cache-Control: `private, no-cache, no-store`** for sensitive responses
- **JWT**: encoded, NOT encrypted; must verify signature; beware `alg: none` attack

### 🍪 Session Management
- Session ID: minimum **128 bits**, generated by **PRNG**
- **Never** put session IDs in URLs, logs, or referrer headers — use cookies
- Cookie attributes: **HttpOnly** (blocks JS access), **Secure** (HTTPS only), **SameSite** (CSRF)
- **Invalidate session server-side** on logout — don't just clear client cookie
- Create a **new session** after privilege change (e.g., pre-login → post-login)
- Set **inactivity timeout** proportional to risk; provide visible logout
- Allow users to view/terminate all active sessions

### 🔐 Data in Transit
- Use **HTTPS everywhere** — not just on login/payment pages
- **HSTS** eliminates insecure initial HTTP request window — enable only when 100% HTTPS ready
- Get certificate from trusted **CA** (DV → OV → EV, increasing trust/cost)
- Set `Secure` flag on cookies; prevent sensitive data in URLs
- Test TLS config monthly with tools like **SSLTest (immuniweb.com)**
- Keep **private key secret** — protect it like a password

### 💥 XSS / CSRF / SSRF
| | XSS | CSRF | SSRF |
|--|-----|------|------|
| **Who acts** | Victim browser | Victim browser | Server |
| **Prevention** | Escape output, CSP | CSRF tokens, SameSite | Input validation, allowlists |

- **Reflected XSS**: malicious URL → server reflects → browser executes
- **Stored XSS**: malicious input stored → served to all visitors
- **DOM XSS**: client-side DOM manipulation; server-side validation doesn't help
- **CSP header**: whitelist trusted script/resource origins
- **SSRF** targets: `127.0.0.1`, `169.254.169.254` (cloud metadata), internal APIs

### 🔑 Password Storage
- **Never store plain-text passwords**
- Store: **`{salt, hash, work_factor}`**
- Salt: unique per user; ≥ 32 bits (OWASP), ≥ 128 bits (NIST)
- Algorithm priority: **Argon2id** > scrypt > bcrypt > SHA-256 > ❌MD5/SHA-1
- MD5/SHA-1 are **broken** for passwords — fast = dangerous
- Password requirements: ≥ 12 chars, mixed character types, avoid reuse

### 💉 Injection / SQL
- **Always use parameter binding** — never concatenate user input into SQL
- **Stored procedures** do NOT prevent injection without parameter binding inside them
- **ORMs** do NOT prevent injection if raw query strings are used
- **NoSQL** (e.g., MongoDB `$where`) is also vulnerable to injection
- **Blind SQL injection**: no direct output but behavior reveals data
- **File upload**: validate content, prevent execution, host on CDN/cloud storage

### 📦 OWASP Top-5 Focus (2025)
1. **A01** — Broken Access Control (auth, authz, sessions, transit, SSRF, CSRF)
2. **A02** — Security Misconfiguration
3. **A03** — Software Supply Chain Failures (outdated/vulnerable dependencies)
4. **A04** — Cryptographic Failures
5. **A05** — Injection (SQL, XSS, file upload, etc.)
